2. The three publish-blockers (why you killed the chain)

D1 — Mirror-slayer ran on the 1v1 scenario only ab26a1c

The bug: interactive dives expand to all 9 shield scenarios — but the expansion happened after Phase 2. The slayer iteration, slayer archetypes, and threshold auto-discovery all consumed Phase-2 results, so every published Slayer Builds / Top-Mirror CMP table was computed against 1v1-only mirror fights while everything else on the page is all-9.

Knock-on: at one scenario the S2 graded round metric degenerates to 0/1 wins, so the tie-explosion fix stopped working — the morning chain's log showed a round-1 pool of 1,772 IVs against --mirror-slayer-pool 30.

The fix: expand scenarios at parse time, before anything sims. Bonus: the old flow re-swept the base mode after expansion; that redundant sweep is gone. Smoke dive verified: graded multi-scenario wins, pools of 37/31 against the cap of 30.

L1 — Shadow opponents resolved without shadow multipliers in anchors 92dd629

The bug: the gamemaster's shadow entries ("Quagsire (Shadow)") carry the normal form's base stats — shadowness exists only as a multiplier flag. The anchor resolvers had a shadow parameter that no call site passed and never parsed the name suffix. Every breakpoint anchor vs a shadow opponent was ~20% too strict (def 6/5 too high); every bulkpoint too lenient (atk 1/1.2 too low).

Proof it was live: the published Sylveon dive's card-verified squag_fairy_wind_5 anchor (expert card: atk ≥ 121.70) resolved to ~145.7 and tagged 0 of 78 rows. Post-fix it resolves to 121.44 — the exact formula boundary below the card's first-attainable 121.70.

Same family, focal side: build_auto_anchors didn't take a shadow flag either, so shadow-focal dives (Shadow Sableye/Jumpluff/Ninetales) computed their auto-CMP cohort threshold from unmultiplied attacks (~17% low). Both fixed and pinned by multiplier-ratio tests.

E1 — Obstruct RAISED the opponent's defense d371951

The bug: for buffTarget='both' moves, our buff application used the generic buffs array for both sides. OBSTRUCT (the only such move) has buffsSelf [0,+1] / buffsOpponent [0,-1] — so each Obstruct gave the opponent +1 defense instead of −1, a 2-stage error per throw, compounding to an inverted matchup over a long fight. Two same-root fixes rode along: the selfBuffing flag derivation and the shield-policy sub-route both needed PvPoke's 'both' clauses.

Pinned by: a 9-cell Obstagoon (Counter / Obstruct + Night Slash) vs Azumarill oracle fixture generated from PvPoke's own engine — all 9 match exactly on score, winner, and move log, including a genuine 2v0 tie. (No published impact: Obstagoon is only in the CS tournament pool today.)

3. The engine-fidelity round (oracle-driven)

E2 — The buff meter: a wrong finding hiding a real bug 434fece

The reviewer claimed PvPoke's deterministic chance-buff meter fires on different uses than ours (e.g. Air Cutter on use 4 vs our 3) — which would contradict our passing Corviknight 9/9 fixture. The probe found the reviewer's error: PvPoke initializes the meter to the chance itself (and to 0 for exactly-50% moves), so 0.3 → fires on use 3, matching us… at first. The real divergence: our counter reset after each proc and used round(1/chance); PvPoke's float accumulator never resets, so the schedule isn't periodic. Concretely wrong in our sim before the fix:

ChanceMovesPvPoke procsOurs (old)
0.2Crunch, Shadow Bone4, 10, 14, 195, 10, 15, 20 — first proc late
0.125Night Slash, Cross Poison7, 158, 16 — first proc late
0.3Air Cutter, Scald…3, 6, 10, 13…3, 6, 9, 12…
0.5, 0.33Bulldoze, Zap Cannon…identical — which is why the suite never caught it

Python and JS share IEEE-754 doubles, so the verbatim port reproduces PvPoke's schedule bit-exactly, float-drift quirks included (chance 0.1 procs on use 10 only because ten accumulated 0.1s are still < 1.0). Pinned by parametrized 20-use schedule tests.

Slayer signal-loss: what it is, what S2 already fixed, what the options would do

The problem in one example. On the Oinkologne GL dive, the "slayer" IVs (yellow markers) cluster in the top-RIGHT of the scatter — i.e. they're also the rank-1-ish bulk IVs — instead of the classic Lurgan-Ape pattern where slayers sit LEFT (worse bulk, traded for attack). Mechanism: Oinkologne caps at level ~22 in GL, so the attack range across all 4096 IVs is narrow, and Mud Slap's low damage-per-energy means one extra point of attack rarely crosses a damage tier. The slayer attack breakpoint therefore lands below where premium-bulk IVs already sit — so the "slayer" tag selects nearly everyone and distinguishes nothing. The same saturation shows up wherever an auto-anchor's lowest tier is cleared by ~100% of the cohort.

What the S2 redesign already did about it. The Anchors-First archetype table only COUNTS auto-anchor parents that fewer than 50% of the IV space clears — that's the review's "option 1" (suppress non-differentiating anchors) already applied to the archetype membership. What it did NOT touch: the per-row anchor-tag badges and the Threshold Tiers / Notable-IVs surfaces can still display parents everyone passes, and the 50% gate is membership-only, so a saturated parent still shows in row detail.

The four TODO options, concretely on the Oinkologne scatter:

OptionWhat changes on the pageTrade-off
1. Suppress 100%-cleared sub-anchors in the resolverSaturated tags vanish everywhere (badges, bullets, tiers), not just from archetype membership.Cohort-dependent: two dives with different pools show different anchor sets for the same TOML.
2. Badge only above-median parents per rowRows show only what makes THEM special.Hides "everyone passes 6/6," which is itself information.
3. Rarity-coded badge colorsAll info kept; saturated tags fade visually, rare ones pop.More visual complexity; doesn't shrink the HTML.
4. Move 100%-pass parents to a one-line "everyone clears:" calloutCleanest cards; saturation stated once instead of per-row.Extra renderer state.

My recommendation: 4 + 3 — hoist fully-saturated parents into a single "every IV here clears: X, Y" line per section (so the fact isn't lost, but stops repeating), and rarity-shade the remaining badges. Option 1's resolver-level suppression makes TOML anchors behave differently per pool, which will eventually confuse a re-dive comparison. Test case stays the Oinkologne scatter: success = the yellow markers stop being "everything good" and the remaining tags visibly separate the left-cluster trade-offs.
